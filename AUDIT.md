## Repo location & layout
**Local path:** `~/SayIt`  ·  GitHub `GitCoderAccount/SayIt` (remote `origin`, branch `main`).

No build pipeline — static files served as-is. One `SayIt` class lives in `app.js`; each feature subsystem is a per-file **augmenter** that copies its methods onto `SayIt.prototype`. **Load order** (index.html `<script>` tags): sayit-crypto → core → cache → app → settings → profile → polls → notes → spaces → explore → lists → notifications → channels → threads → bookmarks → banner → embeds → dm.

- `index.html` — HTML + CSS (the UI shell)
- `core.js` — constants, `utils`, `SpaceRTC`, `DMCrypto`, URL/embed parsers, the `CHAINS` registry
- `cache.js` — IndexedDB `Cache` layer
- `app.js` — the `SayIt` class + bootstrap; **`SW_CACHE_VER` lives here**; the feed/render/state/virtualization core + eager wiring + core posting path
- `settings.js · profile.js · polls.js · notes.js · spaces.js · explore.js · lists.js · notifications.js · channels.js · threads.js · bookmarks.js · banner.js · embeds.js · dm.js` — feature augmenters (one subsystem each)
- `boot.js` — pre-paint theme (in `<head>`)  ·  `sayit-crypto.js` — vendored SRI-pinned crypto  ·  `sw.js` — PWA service worker
- `README.md` · `CRYPTO_BUILD.md` · `AUDIT.md` (this file)
- `.ci/` — extractor + ESLint config + `node --test` suite (incl. `augment.test.js`, `render.test.js`) + `smoke.py`  ·  `.github/workflows/` — `lint.yml` (lint+smoke) + `nightly.yml` (live-site smoke)

⚠ Bump **`SW_CACHE_VER`** (top of app.js, `YYYYMMDD-N`) on any app-asset change — CI enforces it. Full architecture / commands / gotchas: the **`reference-sayit-repo`** memory.

- **Boot-order race — THE root cause of the reproducible reload "Startup failed" (2026-06-21).** Symptom: cold first load OK, then a browser reload → feed won't paint + wallet shows disconnected. Proven live with a single clean cache (`sayitdefi-20260621-254` only — no version mixing) yet init still threw `this._noteHTML is not a function` / `this._computeTrends is not a function` during `loadCached → renderFeed → init` (app.js:182), while `typeof pulse._noteHTML === 'function'` *after* load settled. Root cause: `pulse.init()` was called at app.js eval time (app.js:8831), but app.js (`<script>` #3809) loads BEFORE the augmenters that define those methods — notes.js (#3817 → `_noteHTML`), explore.js (#3821 → `_computeTrends`). init paints cached posts inside its async body; on a warm reload its first IndexedDB `await` (getMuted) resolved and rendered before those later scripts ran → augmenter method undefined → init aborts before the background `tryAutoReconnect()` (hence the "disconnected" wallet). Only bites a returning user (cached posts in IDB) on a warm/SW-served reload — which is why cold loads worked and headless smoke (empty IDB → loadCached renders nothing; init error is a *caught* console.error, not a pageerror; `pulse` exists regardless) stayed green. **Fix:** defer boot — wrap the init call in `_bootPulse()` fired on `DOMContentLoaded` (with a `document.readyState` fast-path), so every classic augmenter `<script>` has copied its methods onto the prototype before init renders. Regression guard: `.ci/test/boot-order.test.js` (3 tests — fails if a top-level synchronous `pulse.init(` returns, or the DOMContentLoaded/readyState gate is removed). SW_CACHE_VER → `20260621-255`. 61 unit tests + lint + smoke green. **DEPLOYED `commit d962218`; director real-browser-verified live across 3 warm reloads (feed painted 224→274→324 posts, zero init errors) and OWNER-CONFIRMED (reload works + wallet stays connected).** Lesson: a method called anywhere in init's render path is a BOOT method and must be on the prototype before init runs — either keep it in app.js or defer init until augmenters load (the boot-order rule).
- **Service-worker atomic cache transition (2026-06-21) — hardening; NOT the root cause of the reload crash (that was the boot-order race above — initially misdiagnosed as cache mixing).** Still a real fix: prevents a *different* failure where a deploy could serve a returning visitor a mixed-version asset set. Symptom (originally observed at sayitdefi.com): the feed wouldn't load — console showed `this._noteHTML is not a function` (fatal, from `postHTML`→`renderFeed`→`loadCached`→`init`) and `this._computeTrends is not a function` (Trending panel). `main` was healthy (smoke green; both methods land on the prototype), and the live `app.js`/`SW_CACHE_VER` were current (`20260621-254`) — but **two SW caches coexisted** (`sayitdefi-20260621-253` + `-254`), so a returning visitor's first post-deploy load was served a **mixed-version asset set**: new `app.js` (which calls `_noteHTML`/`_computeTrends`) paired with stale augmenter files from before those methods were split out → boot abort. A manual reload (new SW wins) booted clean. Root cause in `sw.js`: the CACHE_VER switch wasn't atomic — it flipped `currentCacheName` *before* populating the new cache, tolerated *partial* `addAll` failures, and deleted the old cache **unconditionally**. Fix: new `buildCache(name)` helper (same-origin shell = all-or-nothing via transactional `Cache.addAll`; cross-origin ethers CDN = best-effort `allSettled`); the version switch now **builds the new cache fully first**, and only then flips `currentCacheName` + deletes the old — on any shell-fetch failure it **discards the partial new cache and keeps the old, known-good one** (retries next load). `install` routed through the same helper. New regression test `.ci/test/sw.test.js` (4 tests: silent first-adopt, failed-upgrade-keeps-old, recovery-after-failure, all-shell-assets-cached) drives the handler in a sandboxed mock SW. **sw.js-only change → no `SW_CACHE_VER` bump** (not in the lint.yml guard's 20-file list; browsers auto-detect the sw.js byte change). The lingering `253` cache self-cleans on the new SW's `activate` (deletes caches ≠ current). 58 unit tests + lint + `node --check sw.js` + smoke all green. **DEPLOYED `commit 072e1e2`; verified live (new SW activated, caches collapsed to a single `255` after the boot-order deploy).** Prevents the mixed-version boot recurring on future deploys; returning users already broken are fixed by one reload (can't be force-cleared remotely).
